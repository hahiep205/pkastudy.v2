# Supabase Migration Blueprint: Phases 1-3

## Scope

This document locks the implementation decisions for the first three migration phases:

1. Architecture and migration standards
2. Supabase Postgres schema, indexes, and triggers
3. Supabase Auth cutover while preserving the current Express API shape

## Target Architecture

- Frontend remains React/Vite.
- Authentication moves to Supabase Auth.
- Express remains the application orchestration layer for the current project.
- Postgres in Supabase becomes the long-term system of record.
- Supabase Storage becomes the long-term media store.
- Transaction-heavy workflows move to SQL functions/RPC later; they are not part of phases 1-3.

## Identity Strategy

- `auth.users.id` is the canonical user identity.
- User-owned tables in the new schema use `uuid` foreign keys that reference `auth.users(id)`.
- Catalog tables keep integer identities for efficient ordering and simpler admin import/export:
  - `courses`
  - `topics`
  - `flashcards`
  - `toeic_tests`
  - `toeic_question_groups`
  - `toeic_questions`
- `profiles.legacy_user_id` is retained to support staged migration from the current MySQL `Users.id`.

## Why Express Stays For Now

Keeping Express during phases 1-3 gives the current system the smoothest path forward:

- frontend API contracts remain stable
- admin and validation logic do not need a big-bang rewrite
- we can cut Auth over before rewriting every model
- we can continue to gate privileged operations on the server

## Data Ownership Rules

- Public read data:
  - `courses`
  - built-in `topics`
  - built-in `flashcards`
  - `toeic_tests`
  - `toeic_question_groups`
  - `toeic_questions`
- User-owned data:
  - `profiles`
  - `user_progress`
  - custom `topics`
  - custom `flashcards`
  - `user_word_progress`
  - `srs_reviews`
  - `toeic_test_records`
  - `support_tickets`
  - `vocab_activity_logs`

## Migration Standards

- Runtime `CREATE TABLE IF NOT EXISTS` logic should be removed over time.
- All Supabase database changes must live in SQL migration files under `supabase/migrations`.
- Every mutable table gets:
  - `created_at timestamptz not null default now()`
  - `updated_at timestamptz not null default now()`, maintained by a trigger
- MySQL `TINYINT(1)` becomes `boolean`.
- MySQL `JSON` becomes `jsonb`.
- MySQL `AUTO_INCREMENT` becomes `generated by default as identity`.
- MySQL enums become `text` plus `check` constraints unless a dedicated Postgres enum clearly improves correctness.

## Auth Cutover Rules

- Frontend signs in with Supabase directly.
- Frontend exchanges the Supabase access token with the existing backend session endpoints.
- Backend verifies Supabase JWTs and maps them to the current local user records.
- During the transition, the backend still reads domain data from the current database layer.
- Default local demo/admin accounts are mirrored into Supabase Auth when configuration is present.

## Deliverables Completed In Phases 1-3

- architecture and migration blueprint
- first Supabase schema migrations for Postgres
- initial Supabase Auth integration in frontend and backend
- compatibility session endpoints for the current app

## Explicit Non-Goals In Phases 1-3

- no full rewrite of `server/models/*` yet
- no full RLS rollout yet
- no full cutover from disk uploads to Supabase Storage yet
- no removal of Express yet
